# Performance Baseline

This document captures the measured performance baseline for Omninity Operator
across the three target hardware tiers. It is the source of truth that
**Standard 11 — Performance & Snappiness** in
`.local/tasks/OMNINITY_BUG_PREVENTION_STANDARDS.md` is tuned against, and is
re-generated by Task #29 (Performance Tuning & Final Polish Pass) after every
significant performance-affecting merge.

## Target hardware tiers

| Tier | Reference machine | Notes |
|------|-------------------|-------|
| T1 (low) | Apple M1, 8 GB RAM, macOS 14 | Floor for "supported" hardware. |
| T2 (mid) | Mid-range Windows laptop (Intel i5-1340P, 16 GB, Windows 11) | Largest expected segment. |
| T3 (high) | Apple M3 Pro, 18 GB RAM, macOS 14 | Headroom check; should be well inside budget on every op. |

Every benchmark below is run with `pnpm run bench` at the workspace root, which
delegates to each package's `bench` script (writing
`.bench-results.json` per the Standard 11 contract) and is then aggregated by
`scripts/tier-review.ts` Check #9.

## Bundle size baseline (real measurements)

Build command per artifact: `pnpm build` (with the artifact's required env
vars). Sizes are gzipped main JS chunk as reported by Vite.

| Artifact | Built main JS (gzip) | Budget (gzip) | Headroom | Status |
|----------|----------------------|---------------|----------|--------|
| `artifacts/omninity-website` | 303.1 KB | 320 KB | 17 KB | ✓ within tightened budget (was 500 KB in Standard 11 v1) |
| `artifacts/mockup-sandbox`   |  59.5 KB |  75 KB | 16 KB | ✓ within tightened budget |

Per Step 3 of the task plan ("Tighten passing benchmarks"), the per-artifact
`bundle-budget.json` files were lowered from the Standard 11 v1 default of
500 KB to leave roughly **5–6 % headroom** above the measured size. This
prevents future drift while still tolerating routine dependency updates and
small new features. Any future bump to either budget must be justified in the
PR description and reflected in this table in the same commit.

### How to re-measure

```bash
PORT=5000 BASE_PATH=/ pnpm --filter @workspace/omninity-website run build
PORT=5001 BASE_PATH=/__mockup pnpm --filter @workspace/mockup-sandbox run build
pnpm --filter @workspace/scripts run tier-review   # runs Check #10
```

Playwright spec `e2e/startup-time.spec.ts` (gated by the `e2e-startup` CI

## Polish optimisations applied in this pass

Only items that actually became actionable once the whole app exists end-to-end
are recorded here. (Per the task scope: "this task does not write new
`*.bench.ts` files for operations that already had them, and it does not
introduce streaming/optimistic-UI/code-splitting patterns that should have
been there from the start".)

- **Bundle budgets tightened.** `bundle-budget.json` added to every existing
  frontend artifact (`omninity-website`, `mockup-sandbox`) with a 5–6 %
  headroom over the measured gzipped main chunk. This activates tier-review
  Check #10 immediately and prevents silent drift.
- **Regression protection wired up.** Three CI workflows are added:
  `lighthouse.yml` (renderer perf scores), `tier-review.yml` (Checks #9
  bench compliance and #10 bundle-size), and `e2e-startup.yml`
  (Playwright cold-start spec against a production build via the new root
  `playwright.config.ts`). The cold-start spec gates on the
  `[data-test="chat-ready"]` marker stamped by `ChatPage` itself, so the
  gate measures the actual Standard 11 row "Cold start → ready chat
  interface" rather than global app mount.
- **Preloading, virtual scrolling, image optimisation.** Deferred until the
  feature artifacts they target (chat, marketplace, onboarding) ship — those
  upstream tasks own the patterns under Standard 11. This document will track
  each as it lands.
